PASTA Algorithm — Deep Dive

Predictive Adaptive Scheduling with Timed Aging · Three enhancements over pure SJF

Priority Formula

priority = α × norm_sjf + β × norm_aging
SJF Component (α = 0.70)
norm_sjf = 1 − (pred − min_pred) / (max_pred − min_pred + ε)
→ 1.0 for the shortest predicted job, 0.0 for the longest
Aging Component (β = 0.30)
norm_aging = wait_time / (max_wait + ε)
→ 0.0 when just arrived, rises toward 1.0 as job waits longer
In batch mode (all 15 jobs arrive at t=0): wait_time=0 for all → norm_aging=0 → PASTA reduces to pure SJF. In streaming mode, aged jobs get a priority boost, preventing starvation of large jobs.
warning Problem 1 · Starvation

In pure SJF, large jobs can wait indefinitely if short jobs keep arriving. A 10-minute PageRank job might never run.

check_circle PASTA Solution

Aging term: as wait_time grows, priority climbs toward 1.0. Every job is guaranteed to eventually reach the top.

warning Problem 2 · Tier Drift

SJF uses fixed thresholds: small < 10s, large > 60s. On this hardware where all jobs run in 0.1–2s, every job lands in "small" — defeating the purpose.

check_circle PASTA Solution

Dynamic tiers: percentile-based boundaries (p33/p66) adapt to whatever workload is present. Always assigns 3 tiers correctly.

warning Problem 3 · No Learning

SJF uses the model as-is. If predictions are off (e.g., trained on Google Cloud but running locally), errors accumulate and ordering degrades.

check_circle PASTA Solution

Feedback loop: after each run, actual times are appended to raw_metrics.csv and the model retrains. Predictions improve run-over-run.

In local[*] mode (our benchmark): Driver = Worker = same JVM. All CPU cores used as "executors". No network overhead. spark.executor.memory is set at startup and cannot change mid-session. Only spark.sql.shuffle.partitions and spark.scheduler.pool can be changed per-job.

stacked_bar_chart STAGES

Jobs split at shuffle boundaries (groupBy, join, sort). Data must be redistributed across partitions between stages.

Stage 0: map/filter (no shuffle)
Stage 1: reduceByKey (shuffle!)
Stage 2: collect result
task TASKS

Each stage runs one task per partition. More partitions = more parallelism. spark.sql.shuffle.partitions controls this.

4 partitions → 4 tasks/stage
8 partitions → 8 tasks/stage

Jobs A blocks B and C (Convoy Effect). Small jobs wait behind large jobs regardless of urgency.

account_tree FAIR Pools
small_pool: weight 3 → 4 partitions
medium_pool: weight 2 → 8 partitions
large_pool: weight 1 → 16 partitions

PASTA uses FAIR scheduler with 3 pools. Higher-weight pools get more executor slots. Set via fairscheduler.xml.

Why log1p transform?

Execution times are right-skewed — a few long jobs dominate. log1p compresses the range, making predictions more accurate. expm1() reverses it at inference time.
